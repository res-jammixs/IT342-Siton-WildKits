# Database Schema Reference

This document describes the database schema for WildKits. Tables are auto-generated by Hibernate based on JPA entities.

## Tables Overview

| Column | Type | Constraints | Description |
|--------|------|-------------|-------------|
| user_id | BIGSERIAL | PRIMARY KEY | Auto-increment user ID |
| name | VARCHAR(100) | NOT NULL | User's full name |

| account_status | VARCHAR(20) | NOT NULL | PENDING, APPROVED, REJECTED, SUSPENDED |

### 3. products
Stores product listings (items for sale or lend).

| Column | Type | Constraints | Description |
|--------|------|-------------|-------------|
| product_id | BIGSERIAL | PRIMARY KEY | Auto-increment product ID |
| title | VARCHAR(200) | NOT NULL | Product title |
| description | TEXT | | Product description |
| price | NUMERIC(10,2) | NOT NULL | Product price |
| type | VARCHAR(10) | NOT NULL | SELL or LEND |
| status | VARCHAR(20) | NOT NULL | AVAILABLE, SOLD, LENT_OUT, UNAVAILABLE |

**Foreign Keys:**

## Sample SQL Queries

### Get all available products for sale
```sql
SELECT * FROM products 
WHERE status = 'AVAILABLE' AND type = 'SELL'
ORDER BY created_at DESC;
```

### Get user's transaction history
```sql
SELECT t.*, p.title, p.price, u.name as seller_name
FROM transactions t
JOIN products p ON t.product_id = p.product_id
JOIN users u ON p.user_id = u.user_id
WHERE t.user_id = 1
ORDER BY t.transaction_date DESC;
```

### Get pending account approvals
```sql
SELECT aa.*, u.name as user_name, u.email
FROM account_approvals aa
JOIN users u ON aa.user_id = u.user_id
WHERE aa.status = 'PENDING'
ORDER BY aa.approval_date ASC;
```

### Get products with reports
```sql
SELECT p.*, COUNT(pr.report_id) as report_count
FROM products p
LEFT JOIN product_reports pr ON p.product_id = pr.product_id
GROUP BY p.product_id
HAVING COUNT(pr.report_id) > 0
ORDER BY report_count DESC;
```

## Database Migration Notes

**Hibernate DDL Auto Mode**: `update`

This means:
- Tables are automatically created if they don't exist
- New columns are added automatically
- Existing data is preserved
- Columns are NOT removed automatically (must be done manually)

**For Production**:
- Consider using Flyway or Liquibase for version-controlled migrations
- Set `ddl-auto: validate` in production
- Manage schema changes through migration scripts

## Data Validation Rules

### User
- Name: 2-100 characters
- Email: Valid email format, unique
- Password: 6-100 characters (should be encrypted with BCrypt)

### Product
- Title: 3-200 characters
- Description: 10-5000 characters
- Price: > 0, max 8 digits + 2 decimals
- Type: SELL or LEND
- Status: Auto-set to AVAILABLE on creation

### Transaction
- Type: BUY or LEND
- Status: Auto-set to PENDING on creation

### Product Report
- Reason: 10-5000 characters
- Status: Auto-set to PENDING on creation

### Account Approval
- Status: PENDING, APPROVED, or REJECTED
- When approved/rejected, updates user.account_status
